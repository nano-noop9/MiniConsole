# CLAUDE.md

用于减少 LLM 编码常见错误的行为指南。可根据需要与项目特定规范合并。

**权衡：** 本指南倾向于“谨慎优先于速度”。对于简单任务，可自行判断。

---

## 1. 编码前先思考

**不要想当然，不要掩盖疑惑，明确权衡。**

在实现之前：

- 明确说明你的假设；如果不确定，直接提出问题  
- 如果存在多种理解方式，全部列出 —— 不要默认选择其中一个  
- 如果有更简单的方案，直接指出；必要时提出反对意见  
- 如果存在不清晰之处，暂停实现，明确指出问题并提问  

---

## 2. 简单优先

**用最少的代码解决问题，不做任何额外设计。**

- 不实现未被要求的功能  
- 不为一次性代码做抽象  
- 不添加未被要求的“灵活性”或“可配置性”  
- 不为不可能发生的情况写错误处理  
- 如果写了 200 行但可以用 50 行解决，重写  

自检问题：  
“一个资深工程师会认为这段代码过度设计吗？”如果是，继续简化。

---

## 3. 外科手术式修改

**只改必须修改的部分，只清理你自己引入的问题。**

在修改已有代码时：

- 不“顺手优化”无关代码、注释或格式  
- 不重构未出问题的代码  
- 遵循现有代码风格，即使你不认同  
- 如果发现无关的死代码，可以指出，但不要删除  

当你的修改引入“孤儿代码”时：

- 删除你导致未使用的 import / 变量 / 函数  
- 不要删除原本就存在的死代码（除非被要求）  

判断标准：每一行修改都必须能直接追溯到用户需求。

---

## 4. 目标驱动执行

**定义可验证的成功标准，并循环验证直到达成。**

将任务转化为可验证目标：

- “添加校验” → “为非法输入编写测试，并使其通过”  
- “修复 bug” → “先写出复现测试，再让其通过”  
- “重构 X” → “确保重构前后测试均通过”  

对于多步骤任务，先给出简要计划：

1. [步骤] → 验证：[检查方式]


2. [步骤] → 验证：[检查方式]


3. [步骤] → 验证：[检查方式]



强成功标准可以让你独立推进；弱标准（如“让它能用”）会导致反复沟通。

---

**如果这些规则生效，应表现为：**
- diff 中无关修改减少  
- 因过度设计导致的返工减少  
- 在实现前就提出澄清问题，而不是事后修补
